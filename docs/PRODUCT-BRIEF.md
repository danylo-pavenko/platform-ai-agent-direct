## 1. Коротко про продукт

**Direct AI Agents** — це white-label платформа AI-продажів для **Instagram Direct**. Вона підключається до офіційного Meta API, відповідає клієнтам за лічені секунди 24/7, знає каталог бренду, веде живу розмову (не кнопковий скрипт), збирає замовлення або кваліфікує лідів і передає готові картки команді в **Telegram** та **CRM**.

Кожен клієнт отримує **ізольований інстанс**: власна база даних, домен адмін-панелі, окремі процеси на сервері. Дані одного бренду недоступні іншим.

### Ключова цінність

| Біль бізнесу | Що дає платформа |
|---|---|
| Клієнти пишуть уночі та у вихідні — відповідь зранку | Відповідь ~4 с, цілодобово |
| Менеджери витрачають час на «ціна / розмір / доставка» | До ~87% типових діалогів закриває агент без людини |
| Замовлення губляться в переписці | Структурована картка в Telegram + запис у CRM |
| Складно масштабувати підтримку в піки | Один агент обробляє необмежену кількість DM паралельно |
| Різні бізнеси — різні сценарії | Три режими агента: продажі, лідген, запис на послуги |

### Для кого

- **E-commerce** (одяг, косметика, аксесуари) — продаж через DM з каталогом і доставкою.
- **Сервісні бізнеси** (барбершопи, салони, клініки) — запис на послуги, вибір філії, слоти.
- **Агенції та B2B** — кваліфікація ліда, збір пресейл-брифу, передача в CRM.
- **Бренди з високим трафіком у Direct** — зняття рутини з команди без втрати якості комунікації.

## 2. Як це працює (архітектура для бізнесу)

```
Клієнт у Instagram DM
        │
        ▼
   Meta Webhook (офіційний API)
        │
        ▼
   AI-агент (Claude) + системний промпт + знання бренду
        │
        ├──► Відповідь клієнту в Direct
        ├──► Збереження в адмін-панелі (історія, профіль, теги)
        ├──► Telegram-сповіщення команді (замовлення, ескалація, бриф)
        └──► CRM (KeyCRM, CleverBOX, інші за адаптером)
```

**Запуск під ключ — орієнтовно 48 годин:**

1. Підключення Instagram через OAuth (без паролів).
2. Синхронізація каталогу з CRM або ручне наповнення знань.
3. Налаштування тону, правил ескалації, робочих годин.
4. Підключення Telegram-бота для менеджерів.
5. Тест у пісочниці (Sandbox) → go-live.

## 3. Режими агента

Платформа підтримує **мульти-режимний runtime** — один бекенд, різні сценарії під бізнес клієнта.

| Режим | Призначення | Фінальна дія |
|---|---|---|
| **sales** | Інтернет-магазин | Збір замовлення → Telegram + CRM |
| **leadgen** | Пресейл, агенції, B2B | Структурований бриф → Telegram + CRM-лід |
| **booking** | Салони, барбершопи | Запис на послугу → CleverBOX / CRM |

Перемикання режиму — в адмін-панелі (**Налаштування → Режим агента**), без зміни коду.

## 4. Системний промпт — головний інструмент кастомізації

Системний промпт — це **«мозок» агента**. Він визначає, хто агент, як говорить, що може обіцяти, коли ескалює до людини і які бізнес-правила дотримувати. Зміни в промпті не вимагають розробки: достатньо оновити текст і активувати нову версію.

### 4.1. Що керується промптом

| Область | Приклади налаштувань |
|---|---|
| **Tone of Voice** | «ти» / «Ви», сленг бренду, емодзі, довжина відповідей |
| **Сценарії продажу** | Як питати розмір, як пропонувати upsell, політика знижок |
| **Ескалація** | Коли передати людині: скарги, повернення, опт, тиск по ціні |
| **Робочі години** | Поведінка поза графіком: попередження клієнта vs відкладене повідомлення |
| **SLA менеджера** | Скільки годин команда має відповісти після ескалації |
| **Кваліфікація ліда** | Які питання задавати в leadgen-режимі, мінімум полів брифу |
| **Запис на послуги** | Як обирати філію, які послуги комбінувати, політика скасування |
| **Заборони** | Не обіцяти знижки, не коментувати конкурентів, не давати юридичні поради |
| **Сповіщення команді** | *Інструкції агенту, коли ескалювати з конкретною причиною* (див. розділ 6) |

### 4.2. Шари контексту (що бачить агент на кожному повідомленні)

Агент працює не лише з текстом промпту. Бекенд **динамічно збирає** повний контекст:

1. **Активний системний промпт** — версіонований у БД, з можливістю відкату.
2. **Runtime-блок** — поточний час, робочі години, стан розмови (бот / передано менеджеру), профіль клієнта.
3. **База знань бренду** — brand, contacts, delivery, FAQ, categories, services.
4. **Живий знімок каталогу** — синхронізований з CRM (ціни, розміри, **наявність**, позначка «мало (N шт)»).
5. **Профіль клієнта** — телефон, місто, НП, теги, історія замовлень (не питає те, що вже знає).
6. **Кастомні поля CRM** — додаткові питання, які тенант налаштував в адмінці.

### 4.3. Плейсхолдери в промпті

Промпт може містити змінні, які підставляються автоматично:

- `{{WORKING_HOURS_SUMMARY}}` — графік роботи людської команди.
- `{{MANAGER_SLA_HOURS}}` — SLA відповіді менеджера.
- `{{OUT_OF_HOURS_NOTE}}` — текст для клієнта поза графіком.
- `{{BRANCHES_LIST}}` — список філій (режим booking).
- `{{SELECTED_BRANCH}}` — обрана філія в поточній розмові.

### 4.4. Інструменти агента (tools)

Промпт **інструктує**, коли викликати tool; бекенд виконує дію. Набір tools залежить від режиму.

**Спільні (усі режими):**

| Tool | Що робить |
|---|---|
| `update_client_info` | Зберігає ПІБ, телефон, email, місто, відділення НП |
| `tag_client` | Теги для сегментації (vip, hot, repeat_buyer тощо) |
| `classify_intent` | Класифікація наміру (замовлення, скарга, спам…) |
| `request_handoff` | Ескалація до менеджера + **Telegram-сповіщення** |
| `set_conversation_branch` | Фіксація обраної філії (салон) |

**Режим sales:**

| Tool | Що робить |
|---|---|
| `search_catalog` | Живий пошук товару: ціна, розмір, колір, **наявність** |
| `get_delivery_cost` | Вартість доставки Новою Поштою |
| `collect_order` | Створення замовлення → Telegram + CRM |

**Режим leadgen:**

| Tool | Що робить |
|---|---|
| `submit_brief` | Пресейл-бриф → Telegram + CRM-лід |

**Режим booking:**

| Tool | Що робить |
|---|---|
| `search_services` | Пошук послуг у CRM |
| `get_available_slots` | Вільні слоти на дату |
| `book_appointment` | Підтвердження запису в CRM |
| `attach_reference_photo` | Збереження референс-фото від клієнта |

### 4.5. Мета-агент (Teach Chat)

В адмін-панелі є **мета-агент** — AI-редактор промптів. Менеджер описує зміни природною мовою («будь менш формальним», «не пропонуй знижки»), а система пропонує правки до системного промпту. Це прискорює ітерації без залучення розробника.

### 4.6. Sandbox (пісочниця)

Перед go-live можна тестувати агента в інтерфейсі, схожому на Instagram DM: зберегти до 15 тест-кейсів, покроковий replay, перевірка ескалацій і збору замовлення.

| Компонент | Опис |

| **Super Admin** | Центральна панель: тенанти, deploy, webhook hub, health check. |

### 5.5. Додаткові можливості платформи

- **Розпізнавання голосових повідомлень (STT)** — транскрипція voice DM для агента.
- **Обробка shared post** — агент розпізнає товар з поста, який клієнт переслав.
- **Runtime mode Public / Debug** — тест на проді лише з whitelist акаунтів.
- **Імпорт історії DM** — аналіз існуючих розмов після підключення.
- **Аналітика** — time-to-first-response, якість брифу, intent, конверсії.
- **Версіонування промптів** — activate / rollback без деплою.

## 6. Telegram — операційний центр для команди

Telegram-бот — не лише «сповіщалка», а **робочий інструмент відділу продажів**.

### 6.1. Автоматичні сповіщення (вбудовані)

| Подія | Що отримує команда |
|---|---|
| **Нове замовлення** | Картка: товари, сума, контакти, НП, оплата. Кнопки: ✅ Підтвердити / ❌ Відхилити |
| **Ескалація (handoff)** | Причина, останні повідомлення, посилання на діалог. Кнопки: 👤 Взяти / 🤖 Повернути боту |
| **Пресейл-бриф (leadgen)** | Бізнес, ніша, бюджет, контакти, пріоритет hot/warm/cold. Кнопки: 👤 Взяти / 📌 Hot |
| **CRM недоступна** | Повний snapshot замовлення або брифу для ручного введення |
| **Claude: сесія застаріла** | Алерт про необхідність оновити авторизацію |
| **Claude: ліміт використання** | Попередження / вичерпання підписки |
| **Технічні помилки** | Скорочений лог для швидкої реакції |

### 6.2. Сповіщення через системний промпт (гнучка логіка)

Окремого tool «сповістити про закінчення товару» поки немає — але **системний промпт повністю керує поведінкою агента** у таких ситуаціях:

**Приклад правила в промпті:**

> Якщо клієнт питає про товар, якого немає в наявності, або залишилось ≤3 одиниці — запропонуй альтернативу. Якщо це хіт продажів або клієнт готовий купити відсутній розмір — виклич `request_handoff` з причиною: «Запит на відсутній товар: [назва], розмір [X]. Клієнт готовий замовити / чекає наявності».

**Що відбувається:**

1. Агент викликає `search_catalog` або бачить «мало (N шт)» у знімку каталогу.
2. За інструкцією в промпті — ескалює через `request_handoff`.
3. Менеджери отримують **Telegram-картку ескалації** з контекстом діалогу.
4. Менеджер натискає «Взяти» і відповідає клієнту в Direct з адмінки або вручну.

**Інші сценарії, які можна прописати в промпті:**

- Сповістити команду про **оптовий запит** (handoff + тег `wholesale_inquiry`).
- Повідомити про **VIP-клієнта**, який повернувся (handoff urgent + тег `vip`).
- Передати **скаргу на брак** з пріоритетом urgent.
- Зафіксувати **запит на товар, якого немає в каталозі** — для закупівлі.
- Попередити про **велике замовлення** (сума > N грн) — ескалація перед підтвердженням.

Таким чином, **будь-яка бізнес-подія, яку можна описати словами, перетворюється на правило для агента** — без розробки нового коду.

### 6.3. Команди менеджера в Telegram

- `/login` — прив'язка до групи сповіщень.
- `/takeover <id>` — забрати розмову у бота.
- `/return <id>` — повернути боту.
- `/status <id>` — стан розмови.
- `/help` — довідка.

Inline-кнопки на картках дублюють ці дії без набору команд.

### 7.6. Розширення Telegram-сповіщень (продуктовий потенціал)

Функції, які логічно додати поверх поточної архітектури:

| Фіча | Опис |
|---|---|
| **`notify_team` tool** | Пряме сповіщення без ескалації (закінчився товар, аномальний трафік) |
| **Щоденний дайджест** | Топ запитів, втрачені ліди, товари з нульовим залишком |
| **Алерт низького залишку (проактивний)** | Cron після sync: якщо хіт < N шт → TG групі |
| **SLA breach** | Якщо менеджер не взяв handoff за N годин |
| **Відповідь менеджера з Telegram** | Reply у групі → відправка в IG DM |

Більшість із цього вже частково покривається **системним промптом + `request_handoff`**; виділені tools зроблять сценарії зручнішими і зменшать «шум» у групі.

## 9. Безпека та відповідність

- **Офіційний Meta API** — без збереження паролів Instagram.
- **Ізоляція даних** — окрема БД і процеси на клієнта.

## 12. Приклади use case

### Магазин одягу (sales)

Клієнт пересилає пост з худі → агент розпізнає модель → `search_catalog` → пропонує розміри → рахує доставку НП → збирає замовлення → картка в Telegram → менеджер підтверджує одним кліком → замовлення в KeyCRM.

### Барбершоп (booking)

Клієнт питає про стрижку → агент уточнює філію → `search_services` → `get_available_slots` → клієнт обирає час → `book_appointment` → запис у CleverBOX → підтвердження в DM.

### Digital-агенція (leadgen)

Потенційний клієнт пише «потрібен SMM» → агент класифікує intent → збирає нішу, бюджет, контакт → `submit_brief` → бриф у Telegram з пріоритетом hot → лід у CRM → sales дзвонить упродовж SLA.

### Контроль залишків (через промпт)

У промпті: «Якщо залишок ≤3 і клієнт хоче купити — запропонуй альтернативу; якщо наполягає на цьому розмірі — `request_handoff` з причиною "Попит на low-stock SKU"». Команда отримує сигнал для закупівлі або ручного резерву.
